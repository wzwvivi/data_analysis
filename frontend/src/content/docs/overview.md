# 产品介绍

**网络数据分析平台**（Network Data Analysis Platform）面向飞行试验数据的处理与复盘场景，将原本分散的数据接入、解析调度、事件诊断与架次复盘能力整合到同一工作界面。平台以**可追溯、可复用、可协同**为核心设计目标，适用于发布期的稳定交付与长期迭代运维。

## 产品定位

- 构建统一的数据链路，覆盖 **接入 → 解析 → 分析 → 复盘** 四个关键环节。
- 以版本化配置和页面级权限，保证分析口径与访问行为可审计。
- 作为飞行数据的日常工作台，同时具备发布前评审、问题复现与回归验证等多重用途。

## 价值主张

### 一体化工作流

上传解析、任务中心、事件分析、异常检查、工作台联查等模块共享同一身份体系与数据上下文，避免在多个脚本/工具之间往返。

### 版本化配置治理

网络配置与设备协议均支持草稿、审批、发布三态流转，历史任务始终绑定提交时的版本快照，保证解析口径的稳定性。

### 模块化分析能力

- **飞管事件分析**：聚焦飞管系统的事件触发链路。
- **飞控事件分析**：关注飞控侧事件、告警与参数变化。
- **自动飞行性能分析**：面向自动飞行阶段的指标评估。
- **TSN 异常检查**：任务间差异比对，快速收敛问题。

### 架次级联查

试验工作台以架次为核心维度，聚合跨任务数据与指标，支持下钻与横向对比，是日常复盘的首选入口。

### 数据分析说明

平台中的“数据分析”并不局限于结果展示，而是围绕**数据清洗、协议解析、事件抽取、指标比对、异常定位**形成一条连续链路。原始数据上传后，系统会结合所选协议版本与端口范围完成结构化解析，再将结果映射到任务、事件和架次等分析维度中。

在实际使用中，分析人员通常会先从任务结果判断数据是否完整，再进入事件分析模块查看关键触发链路、时间窗口与参数变化，最后结合工作台或架次比对能力完成横向验证。这样的设计可以帮助团队从“看见数据”进一步走到“解释现象”和“定位原因”，更适合飞行试验中的复盘、问题收敛与结论沉淀。

### 运维与治理

管理员模块提供用户与角色管理、构型配置、平台共享数据治理等能力，支持最小权限发布与资产沉淀。

## 适用场景

- **日常分析**：单任务解析、事件定位、指标观察。
- **问题复盘**：跨任务联查、异常对比、回归验证。
- **发布评审**：核心指标趋势、版本变更对照、报告沉淀。
- **团队协作**：配置变更审批、共享数据维护、权限分发。

## 适用角色

- **测试与分析工程师**：承担主链路使用者，完成任务提交、事件定位、结论输出。
- **项目或型号负责人**：跟踪任务进展、把控分析结论、回看变更记录。
- **平台管理员**：维护权限、配置、共享资产及运行策略。

## 设计原则

1. **清晰的工作流**：每一个模块都服务于一条具体的分析路径。
2. **配置即资产**：解析口径、网络配置、协议定义均以版本形式沉淀。
3. **最小权限可见**：默认收敛，管理员按需开放。
4. **长期可追溯**：历史任务与结果长期保留，便于持续复盘。

## 推荐使用路径

1. 通过 **上传解析** 提交原始数据，选择协议版本与端口范围。
2. 在 **任务中心** 跟踪状态，失败可快速回看错误。
3. 根据问题进入对应事件分析模块或 **TSN 异常检查**。
4. 在 **试验工作台** 完成跨任务、跨架次的联查与沉淀。
5. 将结论汇入团队 SOP 或发布材料。
